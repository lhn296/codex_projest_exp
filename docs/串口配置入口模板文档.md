# 串口配置入口模板文档

## 1. 模板用途

这份模板适合这种场景：

- 项目已经有运行时配置结构
- 想做一个最轻量的配置修改入口
- 暂时不想先做 LCD 菜单
- 暂时不想先做 Web 配置页
- 想直接在 `idf.py monitor` 里改参数

当前推荐配合：

- [config_service.c](g:\ESP32_WORK\ESP32_PRO\ESP32_projest_template_codex\Codex_Project_tep\components\services\config_service.c)
- [config_service.h](g:\ESP32_WORK\ESP32_PRO\ESP32_projest_template_codex\Codex_Project_tep\components\services\config_service.h)

一起使用。

## 2. 推荐定位

串口配置入口不建议直接自己管理所有业务逻辑。  
更推荐的结构是：

```text
config_cli_service
-> 只负责收命令、解析命令
-> 调用 config_service

config_service
-> 只负责维护运行时配置
-> 保存 / 读取 NVS
```

这样分层会比较清楚。

## 3. 推荐文件结构

```text
components/services/
  config_service.c
  config_service.h
  config_cli_service.c
  config_cli_service.h
```

## 4. 推荐初始化位置

在主任务里建议顺序是：

```text
display_service_init
config_service_init
config_service_load
config_cli_service_init
wifi/http/ota init
```

也就是：
- 先让配置层准备好
- 再启动 CLI
- 最后再启动依赖这些配置的联网服务

## 5. 推荐主循环位置

在主循环里周期调用：

```c
config_cli_service_process();
```

推荐放在：

- `display_service_process()` 后
- `wifi/http/ota process()` 前后都可以

当前项目里采用的是：

```c
display_service_process();
config_cli_service_process();
wifi_service_process();
http_service_process();
ota_service_process();
```

## 7. 最小模板实现思路

### 第一步：定义 CLI 上下文

```c
typedef struct {
    bool inited;
    char line_buf[256];
    size_t line_len;
} config_cli_service_ctx_t;
```

### 第二步：初始化 UART/VFS

```c
if (!uart_is_driver_installed(UART_NUM_0)) {
    uart_driver_install(UART_NUM_0, 1024, 0, 0, NULL, 0);
}

esp_vfs_dev_uart_use_driver(UART_NUM_0);
```

### 第三步：标准输入改非阻塞

```c
int flags = fcntl(STDIN_FILENO, F_GETFL, 0);
if (flags >= 0) {
    (void)fcntl(STDIN_FILENO, F_SETFL, flags | O_NONBLOCK);
}
```

### 第四步：主循环中逐字符读取

```c
char ch = 0;
ssize_t read_len = read(STDIN_FILENO, &ch, 1);
```

### 第五步：按行执行命令

```c
if (ch == '\n') {
    line_buf[line_len] = '\0';
    execute_line(line_buf);
    line_len = 0;
}
```

## 8. 推荐支持的基础命令

最小建议命令集：

```text
cfg help
cfg show
cfg set wifi <ssid> <password>
cfg set http <url>
cfg set ota <url>
cfg save
cfg load
cfg reset
cfg reboot
```

这组命令已经足够做：

- 查看配置
- 修改配置
- 写入 NVS
- 恢复默认
- 重启验证

## 10. 推荐的 set/save 规则

建议命令行为明确分层：

- `cfg set ...`
  - 只改当前运行时配置
- `cfg save`
  - 才真正写入 `NVS`
- `cfg reboot`
  - 重启后重新加载配置

这样好处是：

- 不会每敲一条 set 都立刻写闪存
- 更安全
- 也方便一次改多个值再统一保存

## 11. 推荐加的保护

### URL 基础校验

对：

- `cfg set http`
- `cfg set ota`

建议至少校验：

- 必须以 `http://` 或 `https://` 开头
- host 不能为空
- host 最好至少包含 `.`，或者是 `localhost`

### NVS 异常值回退

如果启动时从 NVS 读到明显无效的 URL，建议：

- 打一条 `warning`
- 自动回退到 `app_config.h` 默认值

这样可以避免坏配置导致系统不停刷错误。

## 12. 当前模板优点

这套模板的优点是：

- 接入快
- 调试快
- 成本低
- 很适合作为 `v2.0.0` 这种“设备配置化基础版”的第一入口

## 13. 后续可扩展方向

后面如果要继续升级，可以在这套模板基础上扩展：

- `cfg set wifi` 支持带空格密码的更强解析
- `cfg export`
- `cfg import`
- `cfg erase`
- `cfg status`
- `cfg test http`
- `cfg test ota`

## 14. 一句话总结

串口配置入口模板的核心思路就是：

```text
monitor 输入
-> CLI 按字符接收
-> 拼成完整命令
-> 调 config_service
-> save 到 NVS
-> reboot 后重新加载
```

这是一种非常适合 ESP32 早期开发阶段的配置入口方案。
