# v1.9.0 上板验证清单

## 1. 文档定位

这份清单用于验证 `v1.9.0` 的真实 OTA 下载与升级基础链。

本版验证重点不是只看：

- `READY`

而是进一步验证：

- 固件下载
- OTA 分区写入
- 启动分区切换
- 重启进入新版本

---

## 2. 上板前准备

上板前先确认这些条件：

1. 已经使用支持双 OTA 的分区配置
2. 工程可以正常 `idf.py build`
3. `APP_OTA_VERSION_URL` 指向真实可访问的版本接口
4. 版本接口返回的 JSON 中：
   - `version`
   - `url`
   - `message`
   字段完整
5. `url` 指向真实可访问的固件 `.bin`
6. `APP_OTA_AUTO_UPGRADE` 是否按当前验证目标设置正确

建议当前分两轮验证：

### 第一轮

- `APP_OTA_AUTO_UPGRADE = 0`

目标：

- 只验证真实云端版本检查和 `READY`

### 第二轮

- `APP_OTA_AUTO_UPGRADE = 1`

目标：

- 继续验证真实下载与升级

---

## 3. 固件准备检查

在开始真实升级前，先确认：

1. 已生成当前工程的 `.bin`
2. 云端可访问该 `.bin`
3. 浏览器访问固件地址时能正常下载
4. 固件文件来自当前项目，且不是损坏文件
5. 目标版本号高于当前版本号

建议把：

- 当前设备版本
- 云端目标版本
- 固件下载地址

先记录下来，避免测试时混淆。

---

## 4. 编译与烧录

建议命令：

```powershell
idf.py build
idf.py -p COM3 flash
idf.py -p COM3 monitor
```

如果只是修改版本接口地址或升级开关，也建议重新完整烧录一次，避免用到旧配置。

---

## 5. 第一轮验证：版本检查链

在 `APP_OTA_AUTO_UPGRADE = 0` 时，先验证这些：

1. 设备上电后 Wi-Fi 正常联网
2. 串口出现：
   - `wifi state -> GOT_IP`
3. OTA 进入：
   - `CHECK`
4. 如果云端版本高于当前版本，OTA 进入：
   - `READY`
5. 串口日志能看到：
   - `current=...`
   - `target=...`
   - `url=...`
6. LCD OTA 区域显示正常：
   - `OTA : READY`
   - `MSG : ...`

如果第一轮都还不稳定，不建议立刻进入第二轮真实升级。

---

## 6. 第二轮验证：真实下载与升级链

在确认：

- 版本接口正常
- 固件 URL 正常
- 分区配置正常

之后，再把：

- `APP_OTA_AUTO_UPGRADE = 1`

重新编译烧录。

然后重点看这些日志：

1. OTA 是否进入：
   - `DOWNLOADING`
2. 是否看到 OTA 下载字节数相关日志
3. OTA 是否进入：
   - `VERIFY`
4. OTA 是否进入：
   - `SUCCESS`
5. 是否打印：
   - `esp_ota_set_boot_partition`
   - `reboot`
   - 或 `REBOOTING`
6. 设备是否自动重启

---

## 7. 重启后验证

设备重启后，重点确认：

1. 是否能正常再次启动
2. 串口启动头中的版本号是否变成新版本
3. LCD 首页的版本显示是否更新
4. 按键、LED、蜂鸣器、Wi-Fi、HTTP 等原有功能是否仍正常

这一轮非常重要，因为它决定：

```text
升级不仅下载成功，而且真正切换成功
```

---

## 8. 重点串口观察项

建议重点观察这些日志关键词：

- `wifi state -> GOT_IP`
- `OTA_SVC: ota state -> CHECK`
- `OTA_SVC: ota state -> READY`
- `OTA_SVC: ota state -> DOWNLOADING`
- `OTA_SVC: ota state -> VERIFY`
- `OTA_SVC: ota state -> SUCCESS`
- `OTA_SVC: ota state -> FAIL`
- `esp_http_client_read`
- `esp_ota_write`
- `esp_ota_end`
- `esp_ota_set_boot_partition`
- `esp_restart`

---

## 9. LCD 观察项

建议重点观察：

1. OTA 状态区是否能按阶段变化
2. `MSG` 是否能显示：
   - `CHECK`
   - `READY`
   - `REBOOTING`
   - 或失败原因
3. 升级过程中页面是否仍然稳定刷新

---

## 10. 常见失败点排查

### 10.1 只到 READY，不继续升级

先检查：

- `APP_OTA_AUTO_UPGRADE` 是否仍为 `0`

### 10.2 OTA 进入 FAIL

重点看失败原因是否是：

- `OPEN_FAIL`
- `HTTP_STATUS`
- `READ_FAIL`
- `WRITE_FAIL`
- `OTA_BEGIN`
- `OTA_END`
- `SET_BOOT`

### 10.3 升级后重启失败

重点检查：

- 固件 bin 是否正确
- 分区表是否正确
- 目标分区是否有足够空间

### 10.4 版本检查正常，但固件下载地址无效

说明：

- 版本 JSON 正常
- 但 `url` 不可访问或不是正确固件

这类问题要优先去浏览器侧验证真实固件地址。

---

## 11. 当前版本特别提醒

当前这版虽然真实 OTA 主链代码已经接好，但如果云端接口里的：

- `url`

还是测试占位地址，那么：

- 版本检查能成功
- 真实 OTA 下载不会真正成功

所以在第二轮验证前，必须先准备好：

- 真实可访问的 bin 地址
