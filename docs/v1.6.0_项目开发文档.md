# v1.6.0 项目开发文档

## 1. 版本摘要

- 版本号：`v1.6.0`
- 日期：`2026-03-18`
- 平台版本：`ESP-IDF 5.5.3`
- 阶段主题：`HTTP / JSON`
- 版本定位：在 `Wi-Fi` 联网基础上，完成第一条 `HTTP GET -> JSON 解析 -> LCD/日志显示` 的网络应用链路。

---

## 2. 版本目标

本版本重点不是继续验证“能不能联网”，而是正式把项目推进到网络应用层：

- 联网成功后自动发起一次 HTTP 请求
- 接收服务器返回内容
- 解析基础 JSON
- 把结果同步到 LCD 和串口日志
- 沉淀一份后续可复用的 `http_service` 模板

---

## 3. 本版实现内容

### 3.1 新增 HTTP 服务层

新增：

- `components/services/http_service.c`
- `components/services/http_service.h`

当前已具备：

- HTTP 服务初始化
- `GET` 请求发起
- 响应正文缓存
- 基础状态码记录
- 最近一次请求结果摘要缓存
- 自动测试请求逻辑

### 3.2 新增 JSON 基础解析

本版先做轻量解析，主要用于把网络应用主链跑通：

- 优先解析 `slideshow.title`
- 如果没有，再尝试顶层 `title`
- 都没有时回退成 `OK`

目标是先学会：

```text
HTTP 文本响应
-> JSON 解析
-> 提取业务信息
-> 同步到显示层
```

### 3.3 LCD 联动显示升级

`display_service` 本版增加了 `HTTP` 状态区，首页现在可以同时显示：

- `WIFI`
- `IP`
- `RSSI`
- `HTTP`
- `CODE`
- `MSG`

### 3.4 应用主链升级

当前系统主链已经扩展成：

```text
Wi-Fi
-> HTTP
-> JSON Parse
-> display_service / log
```

在 `app_main_task` 中：

- 初始化 `http_service`
- 主循环推进 `http_service_process()`
- 启动日志中打印新的 `HTTP flow`

## 5. 本版本关键开发内容

### 5.1 为什么先做 HTTP，不先做 WebSocket

这版选择先做 `HTTP / JSON`，是因为后续很多能力都建立在它之上：

- OTA
- 配置拉取
- 云端测试接口
- AI 中转接口

相比之下，`WebSocket` 更适合后面做长连接和流式交互时再引入。

### 5.2 为什么把 HTTP 抽成 service

和前面的：

- `wifi_service`
- `display_service`
- `button_service`

一样，HTTP 也被收成独立服务层，目的是：

- 让业务代码不用直接调用 `esp_http_client`
- 后面更容易扩展 `POST / Header / Token / HTTPS`
- 方便后续复用到 `OTA / AI / 云端接口`

### 5.3 为什么 LCD 要显示 HTTP 结果

本版 LCD 不只是显示联网状态，还承担了网络应用层的可视化输出：

- 串口适合看详细日志
- LCD 适合快速看当前状态和结果摘要

这对后面做无串口依赖的设备交互很重要。

## 6. 关键代码归档

### 6.1 HTTP 服务初始化

主要入口：

- `http_service_init()`

职责：

- 初始化状态缓存
- 初始化 LCD 显示默认值
- 准备后续自动请求逻辑

### 6.2 自动请求逻辑

主要入口：

- `http_service_process()`

当前逻辑：

- 只有在 `Wi-Fi GOT_IP` 后才允许发请求
- 默认自动请求一次测试接口
- 成功后不重复自动请求

### 6.3 JSON 摘要提取

主要入口：

- `http_service_parse_json_summary()`

当前定位：

- 先做轻量摘要提取
- 不追求复杂业务协议
- 重点在于把 HTTP 与 JSON 主链跑通

### 6.4 HTTP 正文读取修正

本版调试过程中，曾出现：

- `HTTP : OK`
- `CODE : 200`
- `MSG : JSON_PARSE_FAIL`

但串口又能看到接口确实返回成功的情况。

最终定位原因是：

- 初版读取方式对响应正文处理不够稳
- 容易出现状态码正常，但正文读取与解析链不够可靠的情况

后续修正为：

- 显式 `open`
- 显式读取响应头
- 循环读取正文
- 解析失败时打印错误位置和正文预览

修正后已验证：

- `content_length=429`
- `body_len=429`
- 成功读取 `httpbin.org/json`
- 成功提取 `slideshow.title`

### 6.5 LCD HTTP 显示

主要入口：

- `display_service_show_http_result()`
- `display_service_draw_http_panel()`

显示内容：

- `HTTP`
- `CODE`
- `MSG`

## 7. 硬件与网络前提

本版不新增新硬件，但需要满足：

- 开发板能稳定连上 Wi-Fi
- 无线环境尽量稳定
- 若开发板支持外接天线，联网实验建议优先接好天线

软件前提：

- 在 `app_config.h` 中填写正确的：
  - `APP_WIFI_STA_SSID`
  - `APP_WIFI_STA_PASSWORD`
  - `APP_HTTP_TEST_URL`
