# v1.9.0 开发方案文档

## 1. 版本定位

- 版本号：`v1.9.0`
- 阶段主题：`真实 OTA 下载与升级`
- 平台版本：`ESP-IDF 5.5.3`

本版本的目标，是把 OTA 从：

```text
只做云端版本检查
```

推进到：

```text
真实下载固件
-> 写入 OTA 分区
-> 设置启动分区
-> 重启切换到新版本
```

---

## 2. 本版为什么要做

前面的版本已经把这些基础打通了：

- `v1.5.0`：Wi-Fi
- `v1.6.0`：HTTP / JSON
- `v1.7.0`：OTA 状态骨架
- `v1.8.0`：真实云端版本检查

所以现在最自然的一步，就是把 OTA 从“知道有新版本”推进到“真的把新固件装进去”。

这一步是后面设备长期维护能力的关键。

---

## 3. 本版核心目标

本版计划完成：

1. 调整分区方案，为真实 OTA 留出空间
2. 当云端版本高于当前版本时，下载固件 bin
3. 使用 ESP-IDF OTA API 写入目标 OTA 分区
4. 完成 OTA 写入后设置新的启动分区
5. 重启并切换到新版本
6. 在 LCD 和串口中显示 OTA 过程状态

---

## 4. 本版主链

本版目标主链如下：

```text
Wi-Fi
-> 获取 IP
-> 云端版本检查
-> 比较当前版本和目标版本
-> 下载 firmware bin
-> esp_ota_begin
-> esp_ota_write
-> esp_ota_end
-> esp_ota_set_boot_partition
-> esp_restart
```

---

## 5. 本版重点学习内容

### 5.1 OTA 分区的基本理解

本版必须真正理解：

- 当前运行分区不能直接被覆盖
- 新固件要写入另一个 OTA 分区
- 写完后要切换 boot 分区

### 5.2 固件下载与分块写入

本版会重点接触这条关键链：

```text
esp_http_client_read
-> esp_ota_write
```

也就是：

- 从云端一段一段读取固件
- 一段一段写入 OTA 分区

### 5.3 OTA 状态表达

本版 OTA 状态建议扩充为：

- `IDLE`
- `CHECK`
- `READY`
- `DOWNLOADING`
- `VERIFY`
- `SUCCESS`
- `FAIL`

这样 LCD 和日志才能把升级过程表达清楚。

---

## 6. 本版必须先处理的问题

### 分区空间不足

前面编译时已经出现过分区空间紧张提示。  
这意味着在真实 OTA 开始前，必须优先处理：

```text
partition table
```

否则即使下载逻辑写好了，也可能没有足够空间完成 OTA。

所以本版会把：

- 分区表调整

作为 OTA 真升级的必要前置步骤。

---

## 7. 本版计划改动的模块

预计会重点涉及：

- `components/services/ota_service.c`
- `components/services/ota_service.h`
- `components/services/http_service.c`
- `components/system/app_types.h`
- `components/system/app_config.h`

- `display_service.c`

---

## 8. 本版暂时不做的内容

为了保持主线清晰，本版暂时不做：

- 断点续传
- 复杂回滚策略
- 完整升级进度百分比显示
- 固件签名校验深挖
- 云端完整 OTA 管理后台

本版先把：

```text
成功升级一次
```

作为主要目标。

---

## 9. 预期验证结果

如果本版完成，理想结果应该是：

1. 设备联网成功
2. 云端检查到新版本
3. 串口和 LCD 进入 `DOWNLOADING`
4. 固件写入 OTA 分区成功
5. OTA 状态进入 `SUCCESS`
6. 设置新的启动分区成功
7. 重启后设备运行新版本

---

## 10. 本版意义

`v1.9.0` 的意义很大，因为它会把项目从：

```text
能发现新版本
```

推进到：

```text
能真正安装新版本
```

这一步一旦打通，后面设备就真正具备了远程维护能力。
