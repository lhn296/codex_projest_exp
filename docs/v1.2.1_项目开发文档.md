# Codex Project TEP v1.2.1 版本归档文档

> 版本：`v1.2.1`
> 日期：`2026-03-15`
> 平台版本：`ESP-IDF 5.5.3`
> 阶段主题：`Unified Event Message`
> 版本定位：把 `v1.2.0` 的按键专用消息升级为统一事件消息结构

---

# 1. 版本摘要

`v1.2.1` 是在 `v1.2.0 FreeRTOS Queue` 架构基础上的一次小版本收口。

这版没有新增外设功能，也没有改变用户可见现象，而是把系统内部的事件通信结构进一步工程化：

- 从按键专用消息升级到统一事件消息
- 增加统一事件日志
- 增加基础发送/接收/处理统计

这一版的重点是“让架构更像架构”，而不是“让功能更多”。

---

# 2. 版本目标

本版本核心目标如下：

1. 将 `app_button_msg_t` 升级为统一事件结构 `app_event_msg_t`
2. 为事件消息增加 `source / type / param1 / param2 / timestamp_ms`
3. 让 `button_service` 和 `app_event_task` 都基于统一事件格式工作
4. 增加基础事件统计与更清晰的事件链路日志
5. 保持单击、长按、双击以及 LED 行为不变

---

# 3. 版本背景与演进关系

当前项目在输入与任务通信方向的演进关系如下：

- `v1.1.2`
  - GPIO 下降沿中断
  - 中断唤醒 + 状态机判断
- `v1.2.0`
  - `FreeRTOS Queue`
  - `app_event_task`
  - 事件驱动型结构
- `v1.2.1`
  - 统一事件结构
  - 统一事件日志
  - 基础统计

也就是说：

- `v1.2.0` 解决的是“事件链先跑起来”
- `v1.2.1` 解决的是“事件链先长成工程化的样子”

---

# 4. 本版本核心设计思想

这版最重要的一句话是：

```text
专用消息适合起步，统一消息适合扩展。
```

`v1.2.0` 的消息结构能完成当前按键任务，但如果后面继续接入：

- `XL9555`
- 传感器
- Wi-Fi
- 显示
- AI 状态事件

那么继续维持“按键专用消息”会越来越局限。

所以 `v1.2.1` 先把消息格式统一下来，让当前系统先具备继续扩展的基础。

---

# 5. 本版本实现内容

## 5.1 功能新增

- 新增统一事件来源枚举 `app_event_source_t`
- 新增统一事件类型枚举 `app_event_type_t`
- 新增统一事件消息结构 `app_event_msg_t`
- 增加事件发送成功计数
- 增加事件发送失败计数
- 增加事件接收计数
- 增加事件处理计数

## 5.2 架构调整

- `button_service` 不再发送按键专用消息
- `button_service` 改为发送统一事件消息
- `app_event_task` 不再直接依赖专用按键消息结构
- `app_event_task` 开始先按统一事件头做分发

## 5.3 行为保持不变

当前用户可见行为与 `v1.2.0` 一致：

- `SHORT`
  - `OFF -> ON -> BLINK_SLOW -> BLINK_FAST -> OFF`
- `LONG`
  - 任何状态直接 `OFF`
- `DOUBLE`
  - 任何状态直接 `BLINK_FAST`

# 6. 软件结构归档

本版的核心结构如下：

## 6.1 `bsp_button`

继续负责：

- GPIO 输入
- GPIO 中断
- 原始电平读取

## 6.2 `button_service`

负责：

- 按键状态机识别
- 产生统一事件消息
- 发送事件到队列
- 记录发送统计

## 6.3 `app_event_task`

负责：

- 阻塞等待队列消息
- 解析统一事件头
- 按类型做业务分发
- 记录接收和处理统计

## 6.4 `led_service`

继续负责：

- LED 模式管理
- 闪烁执行

## 6.5 `app_main_task`

负责：

- 创建统一事件队列
- 初始化服务
- 启动事件任务
- 驱动主循环

# 7. 本版本关键开发内容

## 7.1 从专用消息升级到统一消息

本版最关键的变化就是：

从：

```c
app_button_msg_t
```

升级到：

```c
app_event_msg_t
```

这让消息格式从“只适合按钮”变成“可以继续扩展到其他模块”。

## 7.2 统一事件头设计

本版统一事件结构中最重要的字段是：

- `source`
- `type`
- `param1`
- `param2`
- `timestamp_ms`

当前版本约定：

- `param1`
  - `button_id`
- `param2`
  - `button_event`

这是一种典型的“先统一格式，再逐步扩展语义”的做法。

## 7.3 button_service 的职责进一步清晰

`button_service` 在这版里的定位更明确了：

- 不直接处理 LED 业务
- 不关心谁消费事件
- 只负责识别事件并发送统一消息

这一步很重要，因为它让输入层真正和业务层脱开了。

## 7.4 app_event_task 开始具备“统一分发器”雏形

这版里 `app_event_task` 已经开始按统一事件头判断：

- `source`
- `type`

虽然当前仍然只处理 `BUTTON` 分支，但结构上已经具备以后继续扩展的入口。

这正是后面做：

- 传感器事件
- 网络事件
- UI 事件

的前置基础。

## 7.5 增加可观察性

本版加的统计虽然不复杂，但意义很大：

- 发送成功次数
- 发送失败次数
- 接收次数
- 处理次数

这能帮助我们开始建立一个重要习惯：

```text
系统不仅要能跑，还要能观察。
```

# 8. 关键代码归档

## 8.1 统一事件结构

```c
typedef struct {
    app_event_source_t source;
    app_event_type_t type;
    int32_t param1;
    int32_t param2;
    int64_t timestamp_ms;
} app_event_msg_t;
```

## 8.2 button_service 发送统一事件

```c
app_event_msg_t msg = {
    .source = APP_EVENT_SOURCE_BUTTON,
    .type = APP_EVENT_TYPE_BUTTON,
    .param1 = (int32_t)btn_id,
    .param2 = (int32_t)event,
    .timestamp_ms = now_ms,
};
```

## 8.3 app_event_task 接收统一事件

```c
app_event_msg_t msg;

if (xQueueReceive(s_event_queue, &msg, portMAX_DELAY) != pdPASS) {
    continue;
}
```

## 8.4 app_event_task 统一分发

```c
if (msg.source == APP_EVENT_SOURCE_BUTTON && msg.type == APP_EVENT_TYPE_BUTTON) {
    app_event_handle_button_message(&msg);
} else {
    ESP_LOGW(TAG, "unhandled event ...");
}
```

# 13. 本版本意义

`v1.2.1` 的意义，不在于功能表面变化，而在于系统结构更能长了。

从这一版开始，项目已经不是“只会传按键消息”，而是开始具备：

- 统一事件头
- 统一事件格式
- 统一事件日志

这为后续继续接入更多模块打下了非常重要的基础。

---

# 14. 下一步建议

这版完成后，后面有两条自然方向：

1. 继续做 `v1.2.x`
   - 完善统一事件系统
   - 比如做事件统计查询、事件类型继续扩展
2. 进入新的功能模块版本
   - 比如 `XL9555`
   - 或其他外设驱动学习

如果从学习 FreeRTOS 的节奏上看，我建议下一步可以先切换到新模块，不必在事件系统上停留过久。
