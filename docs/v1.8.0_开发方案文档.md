# v1.8.0_开发方案文档

## 1. 版本摘要

- 版本号：`v1.8.0`
- 平台版本：`ESP-IDF 5.5.3`
- 版本主题：`云端版本检查基础版`
- 版本定位：把当前 OTA 从“本地配置模拟版本信息”升级成“真实云端版本接口驱动”

---

## 2. 这一版为什么做

到 `v1.7.0` 为止，项目已经具备：

- `Wi-Fi` 联网
- `HTTP GET`
- `JSON` 基础解析
- `OTA` 状态骨架
- LCD 联动显示

但当前 `OTA` 还只是：

```text
本地写死目标版本号
-> 本地比较
-> READY / NO_UPDATE
```

这还不是真正的云端升级链。

所以 `v1.8.0` 的目标就是把这条链推进到：

```text
云端版本接口
-> HTTP 获取版本 JSON
-> ota_service 解析 version / url / message
-> 真实版本比较
-> LCD / 日志显示真实结果
```

---

## 3. 版本目标

这一版建议完成这些核心内容：

1. 接入一个真实的云端版本检查接口
2. 设备通过 `HTTP GET` 拉取真实版本信息 JSON
3. `ota_service` 从云端 JSON 里提取：
   - `version`
   - `url`
   - `message`
4. 用真实云端数据驱动：
   - OTA 状态
   - LCD 显示
   - 串口日志
5. 保持这版先不真正执行固件下载和升级

---

## 4. 这一版不做什么

为了让主线更清晰，这一版先不做：

- WebSocket
- 真实固件 bin 下载
- OTA 分区写入
- 重启切换新固件
- HTTPS 证书校验深挖
- 云端后台管理系统

这版先只解决：

```text
设备如何从真实云端接口获取版本信息
```

---

## 5. 推荐云端方案

当前阶段推荐优先使用：

- `Cloudflare Workers`

原因是：

- 上手快
- 不需要先维护 Linux 服务器
- 很适合做轻量 HTTP JSON 接口
- 对当前版本检查实验足够用

建议后续接口先固定为：

```json
{
  "version": "v1.8.1",
  "url": "https://example.com/codex_project_tep_v1.8.1.bin",
  "message": "new firmware available"
}
```

---

## 6. 推荐分层关系

这一版建议保持并继续强化这条链：

```text
wifi_service
-> http_service
-> ota_service
-> display_service / log
```

其中各层职责是：

- `wifi_service`
  - 负责联网和网络状态维护
- `http_service`
  - 负责访问云端接口并拿到 JSON 文本
- `ota_service`
  - 负责解析版本信息并判断是否存在更新
- `display_service`
  - 负责把 OTA 检查结果显示到 LCD

---

## 7. 推荐接口设计

当前最推荐的版本接口字段：

- `version`
  - 最新版本号
- `url`
  - 固件下载地址
- `message`
  - 给日志和 LCD 用的说明文字

可选但当前先不强制：

- `md5`
- `size`
- `force`
- `build_time`

---

## 8. 推荐实现思路

### 第一步

先把云端接口搭起来，返回固定 JSON。

### 第二步

设备端通过 `http_service` 拉取这个 JSON。

### 第三步

在 `ota_service` 里新增“从 HTTP 结果提取版本信息”的逻辑。

### 第四步

用提取出的：

- `version`
- `url`
- `message`

更新：

- `has_update`
- `ota_state`
- `ota_message`

### 第五步

同步到：

- LCD
- 串口日志

---

## 9. 推荐代码演进方向

### 当前 `v1.7.0`

`ota_service` 还是：

```text
APP_OTA_SERVER_VERSION
APP_OTA_FIRMWARE_URL
```

这种本地配置模拟方式。

### `v1.8.0`

建议改成：

```text
HTTP 获取版本 JSON
-> 解析 version/url/message
-> ota_service_compare_version()
```

也就是说，这一版的重点不是重新写 `http_service`，
而是让 `ota_service` 真正开始消费云端数据。

---

## 10. 对显示层的建议

LCD 这一版建议继续保留：

```text
OTA : IDLE / CHECK / READY / FAIL
MSG : v1.8.1 或 new firmware available
```

后面如果空间允许，也可以继续扩展显示：

- 云端返回版本号
- 当前运行版本号

但当前先不要让页面过于复杂。

---

## 11. 推荐日志输出

这版串口日志建议至少保留这些：

- 云端版本接口 URL
- HTTP 是否成功
- JSON 解析是否成功
- 当前版本号
- 云端版本号
- 比较结果
- 固件 URL

推荐理解方式：

- HTTP 成功不等于 OTA 检查成功
- OTA 检查成功也不等于需要升级

---

## 12. 学习重点

这版最值得学的不是某个 API，而是这些：

- 设备怎么消费一个版本接口
- OTA 为什么通常要分“版本检查”和“真实升级”两步
- JSON 字段怎么映射到项目状态
- 云端接口如何服务设备端逻辑

---

## 13. 推荐验证目标

这一版完成后，至少要验证这些现象：

1. 设备成功联网
2. 设备成功访问云端版本接口
3. 串口能打印出：
   - 当前版本
   - 云端版本
   - 升级地址
4. LCD 能显示：
   - `OTA : READY`
   - 或 `OTA : IDLE / NO_UPDATE`
5. 不需要升级时，不应误判成可升级

---

## 14. 推荐后续衔接

如果 `v1.8.0` 跑通，下一步最自然的是：

- 继续做真实 OTA 下载链

也就是：

```text
云端版本检查
-> 获取固件 URL
-> 下载 bin
-> 写 OTA 分区
-> 重启升级
```

---

## 15. 一句话记住

把 `v1.8.0` 记成一句话就够了：

```text
这一版不是直接升级固件，
而是先让设备真正从云端获取和理解升级信息。
```
