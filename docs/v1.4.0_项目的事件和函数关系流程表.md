# v1.4.0 项目的事件和函数关系流程表

## 1. 版本定位

`v1.4.0` 是当前工程第一次正式进入显示系统阶段。  
这一版的目标不是做复杂界面，而是先把下面这条链做成模板：

- `SPI` 通用总线层
- `LCD_ST7789V` 驱动层
- `BSP LCD` 板级适配层
- `display_service` 显示服务层

这样后面改显示内容时，尽量只调用显示服务接口，不直接去改底层驱动。

    C --> F["led_service<br/><sub>[led_service.c]</sub><br/><sub>（LED 服务）</sub>"]

说明：

- `display_service` 不直接碰 `SPI`，而是通过 `bsp_lcd -> lcd_st7789v -> spi_bus`
- `RST / PWR` 仍由 `XL9555` 控制，所以 LCD 初始化前，`XL9555` 必须已经就绪
- 这版之后，屏幕显示会和 `LED / BEEP / 按键` 一样，成为统一事件链的一个输出端

## 3. 总体初始化流程图

这张图是本版最重要的一张。  
它帮助你理解：为什么显示系统不是单独孤立起来初始化，而是要插进现有工程初始化链里。

```mermaid
flowchart TD

    B --> C["app_main_task()<br/><sub>[app_main_task.c]</sub>"]

    C --> D["xQueueCreate()<br/><sub>[app_main_task.c]</sub><br/><sub>（event_queue 事件队列）</sub>"]
    D --> E["led_service_init()<br/><sub>[led_service.c]</sub><br/><sub>（初始化三路 LED）</sub>"]
    E --> F["beep_service_init()<br/><sub>[beep_service.c]</sub><br/><sub>（初始化蜂鸣器服务）</sub>"]
    F --> G["display_service_init()<br/><sub>[display_service.c]</sub><br/><sub>（初始化显示服务）</sub>"]
    G --> H["bsp_lcd_init()<br/><sub>[bsp_lcd.c]</sub><br/><sub>（LCD 板级初始化）</sub>"]
    H --> I["bsp_xl9555_init()<br/><sub>[bsp_xl9555.c]</sub><br/><sub>（确保 XL9555 可用）</sub>"]
    I --> J["bsp_xl9555_lcd_ctrl_init()<br/><sub>[bsp_xl9555.c]</sub><br/><sub>（PWR / RST 控制脚初始化）</sub>"]
    J --> K["spi_bus_init()<br/><sub>[spi_bus.c]</sub><br/><sub>（初始化 SPI 总线）</sub>"]
    K --> L["lcd_st7789v_init()<br/><sub>[lcd_st7789v.c]</sub><br/><sub>（LCD 初始化序列）</sub>"]
    L --> M["display_service_refresh_home()<br/><sub>[display_service.c]</sub><br/><sub>（绘制首页）</sub>"]

    M --> N["app_event_task_start(event_queue)<br/><sub>[app_event_task.c]</sub>"]
    N --> O["button_service_init(event_queue)<br/><sub>[button_service.c]</sub>"]
    O --> P["主循环 while(1)<br/><sub>[app_main_task.c]</sub>"]
```

说明：

- `display_service_init()` 放在输入服务前面，是为了上电后尽快看到基础状态页面
- `bsp_lcd_init()` 里会依赖 `XL9555` 去控制 `RST / PWR`
- `lcd_st7789v_init()` 只关心屏幕芯片如何初始化，不关心当前项目要显示什么

## 5. 初始化依赖关系图

这张图专门回答一个很常见的问题：  
为什么现在很多模块内部都会“先确保依赖已经初始化”，而不是所有东西都只在最外层初始化一次。

```mermaid
flowchart TD
    A["app_main_task()<br/><sub>[app_main_task.c]</sub><br/><sub>（统一组织启动顺序）</sub>"] --> B["beep_service_init()<br/><sub>[beep_service.c]</sub><br/><sub>（蜂鸣器服务入口）</sub>"]
    A --> C["display_service_init()<br/><sub>[display_service.c]</sub><br/><sub>（显示服务入口）</sub>"]
    A --> D["button_service_init()<br/><sub>[button_service.c]</sub><br/><sub>（按键服务入口）</sub>"]

    B --> E["bsp_xl9555_init()<br/><sub>[bsp_xl9555.c]</sub><br/><sub>（需要 XL9555 控蜂鸣器）</sub>"]
    C --> F["bsp_lcd_init()<br/><sub>[bsp_lcd.c]</sub><br/><sub>（需要 LCD 板级初始化）</sub>"]
    D --> G["bsp_button_init()<br/><sub>[bsp_button.c]</sub><br/><sub>（需要板载按键底层）</sub>"]

    F --> E
    G --> E

    E --> H["xl9555_init()<br/><sub>[xl9555.c]</sub><br/><sub>（XL9555 驱动入口）</sub>"]
    H --> I["i2c_bus_init()<br/><sub>[i2c_bus.c]</sub><br/><sub>（XL9555 依赖 I2C 总线）</sub>"]

    F --> J["spi_bus_init()<br/><sub>[spi_bus.c]</sub><br/><sub>（LCD 依赖 SPI 总线）</sub>"]
    J --> K["spi_bus_register_device()<br/><sub>[spi_bus.c]</sub><br/><sub>（注册 LCD SPI 设备）</sub>"]
    K --> L["lcd_st7789v_init()<br/><sub>[lcd_st7789v.c]</sub><br/><sub>（LCD 芯片初始化）</sub>"]
```

说明：

- 现在项目采用的是“上层统一组织初始化 + 下层模块保留依赖自检”的混合方式
- 好处是：`app_main_task()` 里能看到主启动顺序，同时每个模块单独复用时也不容易忘记前置依赖
- 比如 `display_service_init()` 不直接假设 `XL9555` 和 `SPI` 已经全都准备好，而是通过 `bsp_lcd_init()` 把依赖链串起来
- 这也是为什么你会看到某些 `init()` 函数内部先去调用更底层 `init()`，但由于内部都有 `inited` 保护，真正初始化只会发生一次

## 14. 预计关键参数映射表

这一块不是流程图，但很适合理解后面代码里的参数。

### 13.1 LCD 板级参数

- `lcd_rst_pin` -> `APP_XL9555_LCD_CTRL1_PIN`
- `lcd_pwr_pin` -> `APP_XL9555_LCD_CTRL0_PIN`
- `spi_mosi_gpio` -> `GPIO11`
- `spi_miso_or_wr_gpio` -> `GPIO13`
- `spi_sck_gpio` -> `GPIO12`
- `spi_cs_gpio` -> `GPIO21`

### 13.2 显示服务输入参数

- `version` -> 当前项目版本号
- `stage` -> 当前阶段名
- `button_id` -> 最近一次按键编号
- `button_event` -> 最近一次按键事件类型
- `led_id / led_mode` -> LED 状态区显示内容
- `beep_enabled / beep_test_mode` -> 蜂鸣器状态区显示内容

## 15. 哪些地方我觉得还值得额外补
