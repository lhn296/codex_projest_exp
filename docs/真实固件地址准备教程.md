# 真实固件地址准备教程

## 1. 这份文档的定位

这份文档用于解决 `v1.9.0` 进入真实 OTA 升级前最关键的一步：

```text
把当前工程编译出来的 .bin 固件文件
放到一个 ESP32 能访问的公网地址
```

只有这一步准备好，腾讯云版本接口里的：

- `url`

才可以从占位值改成真实固件地址，后面 `ota_service` 才能真正下载和升级。

---

## 2. 当前项目里可直接使用的固件文件

当前工程编译完成后，已经能在：

- `build/codex_project_tep.bin`

看到固件文件。

当前大小大约是：

- `1033152` 字节

这个文件就是后面需要托管出去、供 ESP32 OTA 下载的真实固件文件。

---

## 3. 当前最推荐的托管思路

结合你现在这条主线，我建议优先按下面顺序考虑：

### 方案 A：腾讯云 COS

这是当前最推荐的方案，因为你前面已经在用：

- 腾讯云云函数

做版本 JSON 接口了。

如果继续用腾讯云 COS 托管 bin，会很顺：

```text
腾讯云云函数
-> 返回 version / url / message
-> url 直接指向腾讯云 COS 的 bin 地址
```

优点：

- 国内网络环境一般更稳定
- 和腾讯云版本接口搭配自然
- 浏览器和 ESP32 都更容易验证

### 方案 B：GitHub Release 直链

这个方案也能用，但要注意你当前网络环境是否稳定访问。

优点：

- 简单
- 不需要额外存储服务

缺点：

- 当前环境如果对 GitHub 访问不稳定，ESP32 端也可能不稳

### 方案 C：阿里云 OSS

如果你后面更偏阿里云路线，也可以用。

但就当前已经跑通腾讯云版本接口这一点来说，我更建议继续用腾讯云 COS，减少切换成本。

---

## 4. 当前最推荐的组合

建议当前直接采用：

```text
腾讯云云函数
-> /version 返回 JSON
-> url 指向腾讯云 COS 上的固件 bin
```

最终结构类似：

```json
{
  "version": "v1.9.1",
  "url": "https://your-cos-domain/firmware/codex_project_tep_v1.9.1.bin",
  "message": "new firmware available"
}
```

---

## 5. 固件文件准备建议

在上传前，建议先做这几件事：

1. 重新执行一次：

```powershell
idf.py build
```

2. 确认固件文件存在：

```text
build/codex_project_tep.bin
```

3. 给固件文件起一个更清晰的版本名，建议不要直接长期用默认文件名。

推荐命名方式：

```text
codex_project_tep_v1.9.1.bin
```

或者：

```text
codex_project_tep_v1.9.0_real_ota_test.bin
```

这样后面你在云端看文件时，不容易混淆。

---

## 6. 推荐的实际操作步骤

### 第一步：复制一份带版本号的固件文件

例如把：

```text
build/codex_project_tep.bin
```

复制成：

```text
build/codex_project_tep_v1.9.1.bin
```

这样更适合作为云端正式地址。

### 第二步：上传到公网可访问位置

推荐优先上传到：

- 腾讯云 COS

上传后要确认：

- 浏览器直接打开这个 bin 地址，能开始下载文件

### 第三步：把腾讯云云函数里的 JSON 改成真实地址

把当前版本接口里的：

```json
"url": "https://example.com/codex_project_tep_v1.8.1.bin"
```

替换成真实地址，例如：

```json
"url": "https://your-cos-domain/firmware/codex_project_tep_v1.9.1.bin"
```

### 第四步：浏览器先验证版本接口

先访问：

- 腾讯云版本接口 `/version`

确认看到的是新的真实 URL。

### 第五步：再进入 ESP32 端真实升级测试

这时再考虑把：

- `APP_OTA_AUTO_UPGRADE`

改成：

- `1`

进入真实下载与升级验证。

---

## 7. 固件 URL 验证标准

一个可用的固件下载地址，至少要满足：

1. 浏览器可访问
2. 浏览器点击后能正常下载 `.bin`
3. 不是 HTML 页面
4. 不是需要登录后才能下
5. 设备当前网络环境也能访问

这一步很重要，因为后面如果 OTA 失败，第一怀疑对象往往就是：

- 固件 URL

而不是 OTA 代码本身。

---

## 8. 当前最容易踩的坑

### 8.1 用了占位地址

如果版本接口里还是：

```json
"url": "https://example.com/..."
```

那版本检查能成功，但真实升级一定不会成功。

### 8.2 浏览器能下，不代表 ESP32 一定能下

这个你前面已经在 `Cloudflare Workers` 上踩过一次。

所以 bin 地址准备好后，也要注意：

- 当前 ESP32 所连网络是否能访问

### 8.3 直接覆盖同名文件，后面容易混乱

所以建议：

- 每次版本都带版本号
- 不要长期只用一个 `codex_project_tep.bin` 名字放在云端

---

## 9. 推荐命名方式

建议后面固件统一按这个格式：

```text
codex_project_tep_v1.9.1.bin
codex_project_tep_v1.9.2.bin
codex_project_tep_v2.0.0.bin
```

这样版本接口和云端文件可以直接一一对应。

---

## 10. 当前建议的一句话结论

```text
v1.9.0 进入真实 OTA 前，
最推荐的做法是把 build/codex_project_tep.bin
改成带版本号的 bin 文件，
再上传到腾讯云 COS 这类公网可访问位置，
然后把腾讯云版本接口里的 url 换成这个真实地址。
```
