# v1.7.0 项目开发文档

## 1. 版本摘要

- 版本号：`v1.7.0`
- 日期：`2026-03-18`
- 平台版本：`ESP-IDF 5.5.3`
- 阶段主题：`OTA`
- 版本定位：在 `Wi-Fi + HTTP + JSON` 基础上，完成 OTA 服务骨架、版本比较和 LCD OTA 状态显示。

---

## 2. 版本目标

本版本重点不是一步到位做真实固件升级后台，而是先把 OTA 主链基础骨架搭起来：

- 引入 OTA 服务层
- 跑通版本比较逻辑
- 把 OTA 状态同步到 LCD 和串口
- 为后续真实 OTA 下载与切换固件做准备

---

## 5. 本版本关键开发内容

### 5.1 为什么这版先做 OTA 骨架

因为真实 OTA 完整链路通常会同时涉及：

- 版本检查
- 固件地址获取
- 固件下载
- 分区写入
- 启动切换

如果一口气全做，很容易把学习主线打散。  
所以这版先收成：

```text
版本检查
-> 版本比较
-> OTA 状态流
-> LCD / 串口表达
```

这样更适合当前项目的节奏。

### 5.2 为什么先做版本比较

因为真实设备通常不是每次联网就直接下载固件，而是：

- 先拿服务器版本
- 再和当前版本比较
- 判断是否值得升级

所以这版先把这个思路立住，比直接“盲下载”更有学习价值。

### 5.3 为什么当前默认不自动升级

本版保持：

- `APP_OTA_AUTO_UPGRADE = 0`

目的是：

- 先确保状态链和显示链稳定
- 避免在骨架阶段直接进入不可控升级
- 后面接真实升级下载时更容易定位问题

### 5.4 本版屏幕布局问题修正

调试中发现：

- 左侧 `LED / BEEP` 区局部刷新时，会覆盖右侧 `HTTP / OTA` 区域

最终通过：

- 缩小 `LED / BEEP` 字号
- 重新划分区域
- 让 `HTTP / OTA` 各自拥有独立清屏区

解决了这个问题。

## 6. 关键代码归档

### 6.1 OTA 初始化

主要入口：

- `ota_service_init()`

职责：

- 初始化 OTA 状态
- 初始化显示默认值
- 准备自动检查入口

### 6.2 OTA 状态推进

主要入口：

- `ota_service_process()`

当前逻辑：

- 联网成功后才允许检查
- 依赖 `http_service` 已就绪
- 当前先做一次版本检查

### 6.3 OTA 版本比较

主要入口：

- `ota_service_compare_version()`

当前定位：

- 先做最简单的字符串比较
- 只要当前版本和目标版本不同，就认为有新版本
- 后面可再升级成更严格的语义版本比较

### 6.4 LCD OTA 显示

主要入口：

- `display_service_show_ota_status()`
- `display_service_draw_ota_panel()`

显示内容：

- `OTA`
- `MSG`
