# Wi-Fi掉线调试问题记录

## 1. 问题标题

`v1.5.0` 上板测试时，设备可以连接 Wi-Fi，但运行一段时间后自动掉线，最终进入 `ERROR` 状态。

---

## 2. 现象描述

串口日志初期表现为：

- 可以成功连接到指定路由器
- 可以成功获取 IP
- LCD 和串口都能显示联网成功状态

运行一段时间后出现：

```text
wifi:bcn_timeout,ap_probe_send_start
wifi:ap_probe_send over, reset wifi status to disassoc
WIFI_SVC: wifi state -> DISCONNECTED
WIFI_SVC: wifi disconnected, reason=200
```

后续重连阶段继续出现：

```text
reason=201
```

最终超过重试次数后进入：

```text
WIFI_SVC: wifi state -> ERROR
```

---

## 3. 调试过程

### 3.1 初步怀疑方向

初期怀疑方向主要有：

- Wi-Fi 省电模式导致链路不稳定
- 认证配置兼容性问题
- 路由器距离较远导致信号偏弱
- 开发板无线接收条件较差

### 3.2 增加调试手段

为了更快定位问题，在代码中增加了：

- 断线原因码日志打印
- 关闭 Wi-Fi 省电模式
- LCD 显示 `RSSI`
- LCD 显示 `channel`

这样后面可以更快判断：

- 是连接阶段失败
- 还是连接成功后运行阶段掉线
- 是代码逻辑问题
- 还是无线环境问题

### 3.3 关键日志判断

后续日志显示：

- `reason=200`
  - 对应 `WIFI_REASON_BEACON_TIMEOUT`
- `reason=201`
  - 对应 `WIFI_REASON_NO_AP_FOUND`

同时实测发现：

- 板子距离路由器较远
- 当前开发板未接外部天线
- `RSSI` 偏弱，曾出现 `-77 dBm`

---

## 4. 最终结论

这次问题最终确认：

- **主因是无线环境问题，不是 Wi-Fi 主流程代码错误**

具体表现为：

- 路由器距离较远
- 开发板未接天线
- 接收信号偏弱
- 导致运行过程中出现 `beacon timeout`
- 掉线后重连阶段又因为信号条件差，出现 `no ap found`

当把开发板放到更合适的位置、信号条件改善后，连接恢复稳定。

---

## 5. 本次结论沉淀

后续遇到 Wi-Fi 不稳定问题时，建议优先按这条顺序排查：

1. 先看是否已经成功拿到 IP
2. 再看断线原因码
3. 再看 `RSSI`
4. 判断是代码逻辑问题，还是无线环境问题

当前版本里最有价值的排查信息包括：

- `wifi_state`
- `disconnect reason`
- `RSSI`
- `channel`

---

## 6. 经验记录

### 6.1 当前阶段优先结论

- 如果已经能成功连接并拿到 IP，说明主链基本是通的
- 后面再掉线，不要第一时间只怀疑代码
- 要优先结合 `reason` 和 `RSSI` 判断无线环境

### 6.2 对后续实验的建议

- 联网实验尽量靠近路由器
- 优先使用稳定的 `2.4G`
- 如果开发板支持外接天线，后续联网主线建议尽量接好天线
- 在屏幕上显示 `RSSI` 很有价值，能直接帮助判断环境问题
