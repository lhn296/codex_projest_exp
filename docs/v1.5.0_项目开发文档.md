# v1.5.0 项目开发文档

## 1. 版本摘要

- 版本号：`v1.5.0`
- 日期：`2026-03-18`
- 平台版本：`ESP-IDF 5.5.3`
- 阶段主题：`Wi-Fi Network Foundation`
- 版本定位：在 `v1.4.1` 的显示模板基础上，正式接入 `Wi-Fi STA` 联网主线，建立网络状态管理、事件驱动联网流程和 LCD 联网状态显示模板。

## 2. 版本目标

- 跑通 `Wi-Fi STA` 联网
- 新增 `wifi_service` 联网服务层
- 把联网状态接入 LCD 和串口日志
- 建立 `Wi-Fi` 基础状态模型
- 为后续 `HTTP / OTA / AI` 打下联网入口基础

## 3. 版本背景与演进关系

- `v1.4.1`
  - 完成 `display_service` 的页面分区与局部刷新
  - 完成 LCD 首页作为系统状态窗口的模板化整理
- `v1.5.0`
  - 在已有 `LCD + Queue + Button + XL9555 + BEEP` 的基础上
  - 正式引入 `Wi-Fi STA`
  - 让项目从“本地交互模板”进入“联网设备模板”

## 4. 本版本核心实现内容

### 4.1 Wi-Fi 服务层

新增：

- `components/services/wifi_service.c`
- `components/services/wifi_service.h`

这一层负责：

- 初始化 `NVS`
- 初始化 `esp_netif`
- 初始化默认事件循环
- 初始化 `esp_wifi`
- 注册 `WIFI_EVENT / IP_EVENT`
- 启动 `STA` 联网
- 管理高层联网状态
- 缓存 `IP / RSSI / channel / disconnect reason`

### 4.2 网络状态模型

本版建立的最小联网状态模型：

- `WIFI_STATE_IDLE`
- `WIFI_STATE_CONNECTING`
- `WIFI_STATE_CONNECTED`
- `WIFI_STATE_GOT_IP`
- `WIFI_STATE_DISCONNECTED`
- `WIFI_STATE_ERROR`

这样后面：

- LCD 状态显示
- HTTP 前置判断
- OTA 启动时机
- AI 网络请求前提

都可以统一建立在这套状态之上。

### 4.3 事件驱动联网主链

本版不是用轮询判断 Wi-Fi 状态，而是用事件驱动方式推进联网流程。

主链是：

```text
wifi_service_start()
-> esp_wifi_start()
-> WIFI_EVENT / IP_EVENT
-> wifi_service_event_handler()
-> wifi_service_set_state()
-> display_service / log
```

这是本版最值得掌握的核心设计点。

### 4.4 LCD 联网信息显示

本版把 Wi-Fi 信息正式接入 LCD 状态区，当前可显示：

- `WIFI`
- `IP`
- `RSSI`
- `channel`

这样后续排查联网问题时，不再只依赖串口日志。

## 6. 本版本关键开发内容

### 6.1 为什么 Wi-Fi 要做成 service

当前项目已经逐步形成：

- `button_service`
- `led_service`
- `beep_service`
- `display_service`

所以联网这一层也继续按相同思路做成：

- `wifi_service`

这样后面：

- `HTTP`
- `OTA`
- `AI`

都可以继续复用这一层，而不是把联网逻辑散落在 `app` 或业务代码里。

### 6.2 为什么要区分 CONNECTED 和 GOT_IP

- `CONNECTED`
  - 只是说明已经连上路由器
- `GOT_IP`
  - 才表示已经通过 DHCP 拿到 IP
  - 后续网络业务真正具备前提

所以本版把：

- “网络真正可用”

建立在：

- `WIFI_STATE_GOT_IP`

而不是只看 `CONNECTED`

### 6.3 为什么同时维护 state 和 EventGroup bits

- `wifi_state_t`
  - 更偏高层业务状态
  - 主要给 LCD 和日志显示

- `EventGroup bits`
  - 更偏 FreeRTOS 任务同步标记
  - 为后续 `HTTP / OTA / AI` 任务等待机制做准备

### 6.4 为什么增加 RSSI 和断线原因显示

Wi-Fi 问题很多时候不一定是代码问题，也可能是无线环境问题。

所以本版把这些信息也接进来了：

- `RSSI`
- `channel`
- `disconnect reason`

这样后面调试时能更快判断：

- 是代码主链问题
- 还是信号/环境问题

## 9. 调试问题归档

### 9.1 Wi-Fi 连接后运行一段时间掉线

本版调试过程中出现过：

- 可以正常联网
- 一段时间后自动掉线
- 最终进入 `ERROR`

最终定位结果：

- **主因是无线环境问题，不是主链代码错误**

具体原因包括：

- 路由器距离较远
- 开发板未接天线
- `RSSI` 偏弱
- 出现 `beacon timeout`
- 掉线后重连阶段又出现 `no ap found`
