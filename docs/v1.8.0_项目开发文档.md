# v1.8.0 项目开发文档

## 1. 版本摘要

- 版本号：`v1.8.0`
- 日期：`2026-03-19`
- 平台版本：`ESP-IDF 5.5.3`
- 阶段主题：`云端版本检查`
- 版本定位：在 `Wi-Fi + HTTP + JSON + OTA骨架` 基础上，把 OTA 版本检查从“本地配置模拟”升级成“真实云端版本接口驱动”。

---

## 2. 版本目标

本版本重点不是直接进入真实固件下载和写分区，而是先把下面这条链跑通：

```text
Wi-Fi
-> HTTP 请求云端版本接口
-> 解析 version / url / message
-> ota_service 比较版本
-> LCD / 串口显示真实云端结果
```

本版目标主要包括：

- 引入真实云端版本接口
- `ota_service` 不再依赖本地写死版本数据
- 跑通 `version / url / message` 的 JSON 解析
- 把真实云端检查结果同步到 LCD 和日志
- 为下一版真实 OTA 下载升级做准备

---

## 3. 本版实现内容

### 3.1 OTA 服务改成真实云端版本接口驱动

本版对：

- `components/services/ota_service.c`

做了核心升级。

当前流程已经从：

```text
本地配置版本号
-> 比较
-> READY / NO_UPDATE
```

升级成：

```text
APP_OTA_VERSION_URL
-> HTTP 请求
-> JSON 解析 version / url / message
-> 比较当前版本
-> READY / NO_UPDATE / FAIL
```

### 3.2 HTTP 服务补齐云端版本检查所需能力

本版对：

- `components/services/http_service.c`
- `components/services/http_service.h`

做了几项关键增强：

- 新增 `http_service_get_response_body()`
- 优化正文读取逻辑，确保完整读取响应体
- 增加 HTTPS 证书包挂接
- 允许上层服务直接拿响应正文做二次 JSON 解析

这一步的意义是：

- `http_service` 不再只是“打印测试 JSON”
- 而是开始真正服务于 `ota_service`

### 3.3 引入真实云端版本接口

本版实际做了两条云端实验路线：

#### 路线 A：Cloudflare Workers

已成功：

- 创建 Worker

- 浏览器访问成功

接口地址已记录在：

- `Cloudflare Workers搭建云端版本接口教程.md`

但在当前 ESP32 所连接的网络环境下，访问 `workers.dev` 地址失败。  
因此本版最终没有把它作为当前正式测试链。

#### 路线 B：腾讯云云函数

腾讯云版本接口已经可以成功返回：

## 5. 本版本关键开发内容

### 5.1 为什么这版先做“真实云端版本检查”

因为真实 OTA 完整链路太长，如果一口气做完，容易把学习点混在一起。  
所以本版先把最关键的第一步做实：

```text
设备如何拿到真实云端版本信息
```

这样后面再进入真实固件下载时，主线会清楚很多。

### 5.2 为什么 `ota_service` 要自己解析 `version / url / message`

因为云端返回的版本信息并不只是一个状态码，而是：

- 目标版本号
- 固件下载地址
- 提示信息

这些内容本质上都属于：

```text
OTA 元数据
```

所以本版开始让 `ota_service` 真正消费云端元数据，而不是只依赖本地配置。

### 5.3 为什么保留 `http_service` 的普通测试 URL

当前配置里仍保留：

- `APP_HTTP_TEST_URL`

它的价值是：

- 用来单独验证 `HTTP / HTTPS / JSON` 主链
- 不让 OTA 版本检查成为唯一网络验证手段

这样调试时能更快区分：

- 是 HTTP 基础链出问题
- 还是 OTA 云端接口出问题

### 5.4 为什么最终改用腾讯云版本接口

因为本版调试中已经明确验证：

- `Cloudflare Workers` 在浏览器端可访问
- 但当前 ESP32 所连接的网络环境下不可访问

而腾讯云版本接口在当前网络环境下可以稳定访问。  
所以为了不让项目主线停住，本版正式验证链最终切到腾讯云。

这也说明一个很重要的工程事实：

```text
浏览器能访问，不代表设备当前网络环境一定也能访问
```

## 6. 关键代码归档

### 6.1 OTA 云端检查入口

主要入口：

- `ota_service_check_once()`

当前职责：

- 请求 `APP_OTA_VERSION_URL`
- 获取 HTTP 正文
- 解析 `version / url / message`
- 比较当前版本
- 更新 OTA 状态和显示信息

### 6.2 云端 JSON 解析

主要入口：

- `ota_service_parse_cloud_payload()`

当前解析字段：

- `version`
- `url`
- `message`

这样后面进入真实 OTA 下载时，`firmware_url` 已经能直接复用。

### 6.3 HTTP 正文读取能力

主要入口：

- `http_service_request_get()`
- `http_service_get_response_body()`

本版这部分非常关键，因为：

- 云端版本检查必须依赖完整正文
- 不能只拿状态码

### 6.4 HTTPS 主链验证

本版还额外验证了：

- 证书包正常
- 普通 HTTPS JSON 请求正常

这一步帮助我们把问题准确定位到：

- `Cloudflare Workers` 当前网络环境不可达

而不是误判成：

- ESP32 不支持 HTTPS
