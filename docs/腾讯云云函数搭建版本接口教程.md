# 腾讯云云函数搭建版本接口教程

> 适用阶段：`v1.8.0` 云端版本检查  
> 目标：在腾讯云上快速搭一个能返回版本 `JSON` 的轻量接口，供 `ESP32` 访问

---

## 1. 这份教程的定位

这份教程不是教你把腾讯云整个平台一次学完，而是只做一件事：

```text
搭一个 /version 接口
-> 浏览器能访问
-> ESP32 能访问
-> 返回 version / url / message
```

当前我们只需要一个最小可用的接口，不追求复杂后台。

---

## 2. 为什么推荐腾讯云云函数

对于当前项目来说，它的优点是：

- 适合做轻量 HTTP 接口
- 国内网络环境一般更容易访问
- 不需要你先维护完整 Linux 服务器
- 后面还能继续扩成：
  - 版本检查接口
  - OTA 元数据接口
  - AI 中转接口

所以它很适合当前 `v1.8.0` 这个阶段。

---

## 3. 当前准备内容

开始前建议准备：

- 一个腾讯云账号
- 已完成实名认证
- 可以登录腾讯云控制台
- 一个简单的版本 JSON 样例

建议当前接口最终返回：

```json
{
  "version": "v1.8.1",
  "url": "https://your-domain/firmware/codex_project_tep_v1.8.1.bin",
  "message": "new firmware available"
}
```

---

## 4. 当前推荐的最小实现思路

当前最推荐的做法是：

```text
腾讯云云函数
-> 建一个 HTTP 接口函数
-> 浏览器访问函数地址
-> 直接返回 JSON
```

你可以把它理解成：

```text
Cloudflare Workers 的国内替代轻量方案
```

---

## 5. 控制台操作建议顺序

### 第一步：进入云函数控制台

在腾讯云控制台进入：

- `云函数 SCF`

建议优先选一个离你近、常用的地域，例如：

- 广州
- 上海
- 北京

---

### 第二步：创建函数

建议创建一个最简单的函数，名字可以类似：

- `codex-version-api`

当前建议：

- 运行环境：先选官方支持且容易写返回内容的运行时
- 代码方式：在线编辑
- 触发方式：选择 HTTP 相关入口

如果控制台界面里看到的是：

- Web 函数
- HTTP 触发
- 函数 URL

这一类都可以，当前目标只有一个：

```text
得到一个浏览器可访问的 URL
```

---

## 6. 建议的最小返回代码

如果你当前选的是 JavaScript / Node.js 运行时，可以先按这种最小思路写：

```javascript
exports.main_handler = async (event, context) => {
  const body = {
    version: "v1.8.1",
    url: "https://your-domain/firmware/codex_project_tep_v1.8.1.bin",
    message: "new firmware available",
  };

  return {
    statusCode: 200,
    headers: {
      "Content-Type": "application/json; charset=utf-8"
    },
    body: JSON.stringify(body)
  };
};
```

这段的核心作用就是：

- 收到请求
- 直接返回一个 JSON

当前还不需要数据库，也不需要复杂逻辑。

---

## 6.1 腾讯云 Web 函数直接粘贴版

如果你当前走的是：

- `Web函数`
- `Nodejs`
- `在线编辑`
- `监听端口 = 9000`

那更推荐直接按下面这套最小可用配置来做。

### `app.js`

```javascript
const http = require('http');

// 这里定义版本接口返回的数据。
// 当前先写成固定内容，后面如果需要再改成动态版本管理。
const versionInfo = {
  version: 'v1.8.1',
  url: 'https://example.com/codex_project_tep_v1.8.1.bin',
  message: 'new firmware available'
};

// 创建一个最小 HTTP 服务。
// 当前只做两个路径：
// 1. 根路径：用于检查服务是否启动成功
// 2. /version：返回 ESP32 需要的版本 JSON
const server = http.createServer((req, res) => {
  if (req.url === '/') {
    res.writeHead(200, {
      'Content-Type': 'text/plain; charset=utf-8'
    });
    res.end('codex version api is running');
    return;
  }

  if (req.url === '/version') {
    res.writeHead(200, {
      'Content-Type': 'application/json; charset=utf-8',
      'Cache-Control': 'no-store'
    });
    res.end(JSON.stringify(versionInfo, null, 2));
    return;
  }

  // 其他路径统一返回 404，方便排查请求路径是否写错。
  res.writeHead(404, {
    'Content-Type': 'text/plain; charset=utf-8'
  });
  res.end('Not Found');
});

// 这里必须监听控制台页面里配置的端口。
// 当前教程按 9000 来写，如果你页面改了端口，这里也要一起改。
server.listen(9000, '0.0.0.0', () => {
  console.log('codex-version-api started on port 9000');
});
```

### `package.json`

```json
{
  "name": "codex-version-api",
  "version": "1.0.0",
  "private": true,
  "scripts": {
    "start": "node app.js"
  }
}
```

### 为什么 `package.json` 也要改

如果保留默认模板，通常会看到类似：

```json
"start": "node ./bin/www"
```

这表示平台启动时跑的是旧模板入口，不一定会执行你新写的 `app.js`。  
所以这里建议直接改成：

```json
"start": "node app.js"
```

这样最直观，也最适合当前这个“只返回版本 JSON”的小项目。

---

## 6.2 当前页面配置建议

当前创建页面建议保持：

- 函数类型：`Web函数`
- 运行环境：`Nodejs 20.x` 这类稳定版本优先
- 提交方法：`在线编辑`
- 监听端口：`9000`
- 公网访问：`启用`
- 内网访问：`启用`

其中最关键的是：

```text
公网访问必须启用
```

否则浏览器和 `ESP32` 都没法从公网访问这个版本接口。

---

## 7. 接口路径建议

如果控制台支持直接配置访问路径，建议就做成：

```text
/version
```

如果当前平台给你的就是一个完整函数 URL，没有额外路径也没关系。  
那就直接让这个地址本身返回版本 JSON。

也就是说，后面 `ESP32` 里配置的地址可能是两种之一：

### 方式 A

```text
https://xxx.tencentcloudapi.com/version
```

### 方式 B

```text
https://xxx.tencentcloudapi.com/
```

只要返回内容是版本 JSON 就可以。

## 8. 浏览器验证标准

部署完成后，先不要急着上板，先用浏览器打开地址验证。

期望看到：

```json
{
  "version": "v1.8.1",
  "url": "https://your-domain/firmware/codex_project_tep_v1.8.1.bin",
  "message": "new firmware available"
}
```

如果浏览器都打不开，`ESP32` 也一定访问不了。

---

## 9. ESP32 端如何接入

浏览器验证通过后，把工程里的：

```c
APP_OTA_VERSION_URL
```

改成你腾讯云云函数实际可访问的版本接口地址。

然后主链就是：

```text
Wi-Fi
-> http_service 请求腾讯云版本接口
-> ota_service 解析 version / url / message
-> LCD 显示 OTA 状态
```

---

## 10. 当前阶段最重要的检查点

你后面验证时，最重要的是这 4 点：

1. 浏览器能访问版本接口
2. 返回内容真的是 `application/json`
3. JSON 里至少有：
   - `version`
   - `url`
   - `message`
4. ESP32 当前网络环境也能访问这个地址

第四点很重要，因为我们前面已经踩过一次：

```text
浏览器能访问
不代表 ESP32 当前网络一定也能访问
```

---

## 11. 当前最推荐的返回格式

建议长期固定成：

```json
{
  "version": "v1.8.1",
  "url": "https://your-domain/firmware/codex_project_tep_v1.8.1.bin",
  "message": "new firmware available"
}
```

这样好处是：

- 当前 `ota_service` 直接可复用
- 后面切换平台时，设备端解析逻辑基本不用变
- 后面进入真实 OTA 下载时，也不用再重做版本接口结构

---

## 12. 当前不建议一开始就做的东西

这一步先不要上来就折腾：

- 鉴权签名
- 多环境发布
- 数据库存版本
- 平台化设备管理
- 自动上传固件

当前只要先把：

```text
固定返回版本 JSON
```

做通，就已经足够推进 `v1.8.0`。
