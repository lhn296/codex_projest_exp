# v2.0.0 开发方案文档

## 1. 版本定位

- 版本号：`v2.0.0`
- 阶段主题：`设备配置化基础版`
- 核心目标：把当前项目中写死在 `app_config.h` 里的关键联网参数，升级成“可保存、可读取、可恢复默认值”的运行时配置。

这个版本不是继续堆新的联网能力，而是把已经做好的能力变成更像“产品”的形态：

```text
Wi-Fi
HTTP
OTA
```

这些能力当前都已经具备，但它们依赖的很多参数还是：

```text
写死在代码里
```

`v2.0.0` 要解决的，就是这一步。

---

## 2. 为什么这一版适合做成 v2.0.0

前面的主线已经完成了这些阶段：

- `v1.5.0`：Wi-Fi 联网基础
- `v1.6.0`：HTTP / JSON
- `v1.7.0`：OTA 状态骨架
- `v1.8.0`：真实云端版本检查
- `v1.9.0`：真实 OTA 下载与升级

到了这里，项目已经不缺“能不能联网、能不能升级”的基础能力了。  
现在最缺的是：

- 参数能不能不写死
- 设备重启后能不能保留配置
- 后面做配网、云端切换、批量部署时能不能更方便

所以这一版适合从“模板工程”走向“可配置设备”。

---

## 3. 版本目标

本版建议先完成这些内容：

1. 新增 `config_service`
- 负责管理设备关键配置
- 负责默认值加载、NVS 读取、NVS 保存

2. 把几类关键参数从“纯宏定义”升级成“运行时配置”
- Wi-Fi SSID
- Wi-Fi 密码
- OTA 版本接口地址
- HTTP 测试地址

3. 支持默认值回退
- 当 NVS 中没有保存值时，自动使用默认配置

4. 支持配置读取接口
- 供 `wifi_service`
- `http_service`
- `ota_service`
- `display_service`
查询当前配置

5. 支持基础配置写入接口
- 当前版本可以先通过代码调用测试
- 后面再接按键菜单 / 串口命令 / 配网页面

---

## 4. 本版暂不做的内容

这一版先不做太重的东西：

- Web 配网
- 蓝牙配网
- 手机 App 配置
- 云端远程改配置
- 配置加密存储
- 多套配置切换

先把：

```text
默认值
-> NVS 保存
-> 上电加载
-> 服务层使用
```

这条基础链跑通。

---

## 5. 推荐新增模块

建议新增：

- `components/services/config_service.c`
- `components/services/config_service.h`

它的职责建议是：

- 初始化 NVS 配置命名空间
- 维护一份运行时配置结构体
- 提供默认配置装载
- 提供保存和读取接口

建议不要把这些逻辑继续塞进：

- `wifi_service`
- `ota_service`
- `http_service`

因为这些模块更适合只关心“使用配置”，而不是“保存配置”。

---

## 6. 推荐配置结构

建议先定义一份设备配置结构体，例如：

```c
typedef struct {
    char wifi_ssid[33];
    char wifi_password[65];
    char http_test_url[160];
    char ota_version_url[160];
} app_runtime_config_t;
```

这版先把最关键的联网参数纳入即可。  
后面再扩：

- 设备名
- 上报地址
- 音量
- 屏幕亮度
- 其他业务参数

---

## 7. 推荐默认值来源

当前不建议一下把 `app_config.h` 废掉。  
更推荐把它变成：

```text
默认值来源
```

例如：

- `APP_WIFI_STA_SSID`
- `APP_WIFI_STA_PASSWORD`
- `APP_HTTP_TEST_URL`
- `APP_OTA_VERSION_URL`

保留这些宏，但意义改成：

```text
首次启动或恢复默认配置时使用的默认值
```

这样好处是：

- 兼容当前项目结构
- 后面接配置化时改动不会太剧烈

---

## 8. 推荐初始化顺序

本版建议主初始化顺序变成：

```text
display_service_init
-> config_service_init
-> config_service_load
-> wifi_service_init
-> http_service_init
-> ota_service_init
```

核心思想是：

```text
先把配置读出来
再让联网服务开始使用这些配置
```

而不是让每个服务自己再去读一遍 NVS。

---

## 9. 推荐改造点

### 9.1 wifi_service

当前是直接使用：

- `APP_WIFI_STA_SSID`
- `APP_WIFI_STA_PASSWORD`

本版建议改成：

- 从 `config_service_get()` 取得当前 SSID / 密码

### 9.2 http_service

当前使用：

- `APP_HTTP_TEST_URL`

本版建议改成：

- 从 `config_service_get()` 读取测试 URL

### 9.3 ota_service

当前使用：

- `APP_OTA_VERSION_URL`

本版建议改成：

- 从 `config_service_get()` 读取 OTA 版本接口地址

### 9.4 display_service

可以新增一个简单配置显示区，或者在调试页显示：

- 当前 Wi‑Fi SSID
- 当前 OTA URL 是否来自默认配置 / NVS 配置

---

## 10. 推荐对外接口

建议 `config_service` 至少提供：

- `config_service_init()`
- `config_service_load()`
- `config_service_save()`
- `config_service_reset_to_default()`
- `config_service_get()`
- `config_service_set_wifi()`
- `config_service_set_urls()`

这样前期够用，后面也好扩展。

---

## 11. 本版验证重点

本版建议重点验证这几件事：

1. 首次上电时
- 没有 NVS 配置
- 能正确加载默认值

2. 写入新配置后
- 重启仍能保留

3. Wi‑Fi 改成从配置读取后
- 仍能正常联网

4. OTA 版本接口改成从配置读取后
- 仍能正常做版本检查

5. 恢复默认值后
- 设备能重新回到默认网络参数和默认 URL
