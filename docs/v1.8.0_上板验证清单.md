# v1.8.0 上板验证清单

## 1. 本版验证目标

`v1.8.0` 这版的核心目标不是下载固件，而是验证：

```text
真实云端版本接口
-> HTTP 请求
-> JSON 解析
-> OTA 状态判断
-> LCD / 串口显示
```

也就是说，本版最重要的是把：

- `version`
- `url`
- `message`

这三个字段从真实云端拉回来并正确用起来。

---

## 2. 上板前准备

请先确认这些内容：

### 2.1 Wi-Fi 可正常联网

- 路由器工作正常
- `ESP32` 信号良好
- 当前网络环境可访问你选定的云端版本接口

### 2.2 配置已正确填写

检查：

- `APP_WIFI_STA_SSID`
- `APP_WIFI_STA_PASSWORD`
- `APP_HTTP_TEST_URL`
- `APP_OTA_AUTO_CHECK`
- `APP_OTA_VERSION_URL`

本版正式测试建议：

- `APP_OTA_AUTO_CHECK = 1`
- `APP_OTA_AUTO_UPGRADE = 0`

### 2.3 云端版本接口可先用浏览器打开

在上板前，建议先用浏览器访问：

- 版本接口地址

确认它确实能返回类似：

```json
{
  "version": "v1.8.1",
  "url": "https://example.com/codex_project_tep_v1.8.1.bin",
  "message": "new firmware available"
}
```

---

## 3. 编译与烧录

建议按下面顺序：

```powershell
idf.py build
idf.py -p COM3 flash monitor
```

---

## 4. 启动日志验证

上电后重点看这些日志：

### 4.1 项目版本信息

应看到类似：

- `Version: v1.8.0`
- `Stage: v1.8.0 Cloud Version Check`

### 4.2 Wi-Fi 配置打印

应看到：

- `Wi-Fi config`
- `SSID=...`

### 4.3 OTA 配置打印

应看到：

- OTA 服务初始化完成
- `auto_check=1`
- `version_url=...`

---

## 5. Wi-Fi 主链验证

应依次看到：

- `wifi state -> CONNECTING`
- `wifi state -> CONNECTED`
- `wifi state -> GOT_IP`

并且日志中应看到：

- `wifi got ip=...`
- `rssi=...`
- `channel=...`

LCD 上应能看到：

- `WIFI`
- `IP`
- `RSSI`

---

## 6. HTTP 测试链验证

如果保留了普通测试 URL，本版也应验证：

- `http_service` 能正常访问测试 JSON 接口

串口应看到类似：

- `HTTP_SVC: http status=200`
- `HTTP_SVC: http body={...}`

LCD 上应能看到：

- `HTTP : OK`
- `CODE : 200`

---

## 7. 云端版本接口验证

这是本版最关键的一项。

### 7.1 OTA 进入检查状态

串口应看到：

- `OTA_SVC: ota state -> CHECK`

LCD 上应看到：

- `OTA : CHECK`

### 7.2 云端版本接口返回正文

串口应看到类似：

- `HTTP_SVC: http status=200`
- `HTTP_SVC: http body={ "version": "...", "url": "...", "message": "..." }`

### 7.3 OTA 解析后进入目标状态

如果云端版本高于当前版本，应看到：

- `OTA_SVC: ota state -> READY`

如果版本相同，则应看到：

- `OTA_SVC: ota state -> NO_UPDATE`

如果接口失败，则应看到：

- `OTA_SVC: ota state -> FAIL`

---

## 8. LCD 显示验证

本版重点看 OTA 区是否真实反映云端结果。

### 8.1 期望显示

例如：

- `OTA : READY`
- `MSG : new firmware available`

或者：

- `OTA : NO_UPDATE`
- `MSG : up to date`

### 8.2 需要确认的点

- OTA 区没有被其他区域刷新覆盖
- HTTP 区和 OTA 区能并排稳定显示
- 按键和蜂鸣器状态变化时，OTA 区仍然保持正常

---

## 9. 按键 / 蜂鸣器 / 显示回归验证

虽然本版重点是云端版本检查，但仍建议顺手回归这些功能：

- `KEY0 ~ KEY3` 是否仍正常
- `LED` 状态是否仍正常
- `BEEP / TEST` 状态是否仍正常
- 局部刷新是否仍稳定

这样可以确保本版没有破坏已有功能。

---

## 10. 常见异常与排查方向

### 10.1 浏览器能访问，ESP32 访问失败

优先怀疑：

- 当前设备网络环境不可访问该云端地址

这类问题本版已经实际遇到过，典型是：

- `Cloudflare Workers` 浏览器可访问
- 但当前 ESP32 网络环境不可访问

### 10.2 `HTTP_FAIL`

优先排查：

- URL 是否写错
- 当前网络是否能访问该地址
- 接口是否真返回了正文

### 10.3 `JSON_PARSE_FAIL`

优先排查：

- 返回内容是不是合法 JSON
- 字段名是否是：
  - `version`
  - `url`
  - `message`

### 10.4 OTA 一直 `NO_UPDATE`

优先检查：

- 云端 `version` 是否真的高于当前 `APP_PROJECT_VERSION`

---

## 11. 本版通过标准

满足下面这些，就可以认为 `v1.8.0` 通过：

1. Wi-Fi 正常联网
2. HTTP 测试链正常
3. 真实云端版本接口请求成功
4. OTA 能从云端 JSON 解析出：
   - `version`
   - `url`
   - `message`
5. OTA 状态能进入：
   - `READY`
   - 或 `NO_UPDATE`
6. LCD OTA 区显示正常
