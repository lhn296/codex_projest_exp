# v2.0.0 项目的事件和函数关系流程表

## 1. 文档定位

这份文档专门把 `v2.0.0` 的主线拆成：

- 初始化顺序
- 配置加载关系
- 配置服务与联网服务的依赖关系
- 后续配置写入与保存流程

这一版的核心不是新增联网协议，而是把现有联网能力挂到：

```text
config_service
```

这一层上。

---

## 2. v2.0.0 总体主链

```mermaid
flowchart TD
    A[app_main] --> B[display_service_init]
    B --> C[config_service_init]
    C --> D[config_service_load]
    D --> E[wifi_service_init]
    E --> F[http_service_init]
    F --> G[ota_service_init]
    G --> H[app_main_task loop]
```

这条链的重点是：

```text
先把配置读出来
再初始化联网服务
```

---

## 3. 初始化依赖关系

```mermaid
flowchart TD
    A[app_config.h 默认值] --> B[config_service]
    B --> C[runtime_config]
    C --> D[wifi_service]
    C --> E[http_service]
    C --> F[ota_service]
    D --> G[display_service]
    E --> G
    F --> G
```

这里最关键的变化是：

- `app_config.h`
  - 不再只是最终使用值
  - 更像默认值来源

- `config_service`
  - 成为所有联网相关配置的统一入口

---

## 4. 配置加载流程

```mermaid
flowchart TD
    A[config_service_init] --> B[nvs_flash_init]
    B --> C[config_service_load]
    C --> D{NVS 中有保存配置吗}
    D -- 否 --> E[加载 app_config.h 默认值]
    D -- 是 --> F[从 NVS 读取配置]
    E --> G[生成 runtime_config]
    F --> G
    G --> H[供 wifi/http/ota 查询]
```

---

## 5. 配置保存流程

```mermaid
flowchart TD
    A[业务层修改配置] --> B[config_service_set_xxx]
    B --> C[更新内存中的 runtime_config]
    C --> D[config_service_save]
    D --> E[nvs_open]
    E --> F[nvs_set_str]
    F --> G[nvs_commit]
    G --> H[nvs_close]
```

这条链后面可以来自很多入口：

- 按键菜单
- 串口命令
- 调试函数
- Web 配网
- 云端配置接口

但当前 `v2.0.0` 先不强求入口形式，先把保存链做好。

---

## 6. Wi-Fi 配置使用流程

```mermaid
flowchart TD
    A[wifi_service_start] --> B[config_service_get]
    B --> C[读取 wifi_ssid]
    B --> D[读取 wifi_password]
    C --> E[填充 wifi_config_t]
    D --> E
    E --> F[esp_wifi_set_config]
    F --> G[esp_wifi_start]
```

这一版的关键变化是：

- 不再直接使用 `APP_WIFI_STA_SSID`
- 不再直接使用 `APP_WIFI_STA_PASSWORD`

而是统一从：

- `runtime_config`

读取。

---

## 7. HTTP 配置使用流程

```mermaid
flowchart TD
    A[http_service_request] --> B[config_service_get]
    B --> C[读取 http_test_url]
    C --> D[esp_http_client_init]
    D --> E[open]
    E --> F[fetch_headers]
    F --> G[read body]
```

---

## 8. OTA 配置使用流程

```mermaid
flowchart TD
    A[ota_service_check_once] --> B[config_service_get]
    B --> C[读取 ota_version_url]
    C --> D[http 请求 version.json]
    D --> E[解析 version / url / message]
    E --> F{有新版本吗}
    F -- 否 --> G[IDLE NO_UPDATE]
    F -- 是 --> H[READY]
    H --> I{auto_upgrade 是否开启}
    I -- 否 --> J[等待后续触发]
    I -- 是 --> K[执行真实 OTA]
```

## 10. 推荐运行时结构关系

```mermaid
flowchart TD
    A[app_config.h 默认值]
    B[app_runtime_config_t]
    C[s_wifi]
    D[s_http]
    E[s_ota]

    A --> B
    B --> C
    B --> D
    B --> E
```

这里的意思是：

- `wifi_service`
- `http_service`
- `ota_service`

都不再自己维护一份独立配置源，而是共享：

- `app_runtime_config_t`

---

## 11. LCD 联动建议

这版不建议一开始就在 LCD 上显示太多配置细节。  
先保留最有价值的几个：

- 当前 Wi‑Fi SSID
- 当前 OTA 版本接口是否有效
- 必要时显示“默认配置”或“已保存配置”

推荐思路：

```text
CFG : DEFAULT
SSID: LV-HOME
OTA : URL OK
```

---

## 12. 一句话记住

`v2.0.0` 的流程核心就是：

```text
先用 config_service 把配置统一起来，
再让 Wi‑Fi、HTTP、OTA 都从这份运行时配置读取参数。
```
