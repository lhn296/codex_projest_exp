# v1.4.1 开发方案文档

## 1. 版本定位

`v1.4.1` 是 `v1.4.0` 的显示模板收口版本。  
这一版不引入新的硬件模块，而是继续围绕已经跑通的 LCD 显示链做一次结构和页面层面的整理：

- 优化 LCD 首页布局
- 收紧 `display_service` 的内部结构
- 让显示服务更像后续版本可以直接复用的页面模板

换句话说，这一版的重点不是“让屏亮起来”，而是：

```text
让 LCD 真正成为后续版本统一可复用的信息输出窗口
```

---

## 2. 版本目标

- 优化首页页面布局和视觉分区
- 把 `display_service` 再抽象得更清晰
- 为后续局部刷新预留结构
- 继续保持当前事件链不变
- 让显示模板更适合作为后续所有版本的状态页基底

---

## 3. 本版建议完成内容

### 3.1 页面布局优化

计划把当前首页按功能分成更清晰的区域：

- 标题区
- 版本与阶段区
- LED 状态区
- 蜂鸣器状态区
- 最近按键事件区

目标是：

- 页面阅读顺序更自然
- 每块信息的语义更明确
- 后续增加新状态项时更容易插入

### 3.2 显示服务内部结构优化

计划把 `display_service_refresh_home()` 里较长的绘制逻辑继续拆成几个内部函数，例如：

- `display_service_draw_header()`
- `display_service_draw_project_info()`
- `display_service_draw_led_panel()`
- `display_service_draw_beep_panel()`
- `display_service_draw_last_event_panel()`

这样做的目标是：

- 阅读代码时更容易看懂页面结构
- 后面改单块内容时，不用每次都回到大函数里找

### 3.3 页面参数集中化

计划把当前页面里分散的：

- 坐标
- 颜色
- 字号
- 行距

进一步集中成一组页面布局常量，方便后续：

- 调布局
- 调字号
- 调分区间距

### 3.4 局部刷新结构预留

本版不一定做复杂的脏区刷新，但建议先铺出思路：

- 首页整体刷新保留
- 同时把“按区域绘制”函数拆好
- 后面如果要局部刷新，只需要在这些函数上继续扩展

---

## 4. 本版明确暂不做

为了让 `v1.4.1` 保持聚焦，这一版暂时不做：

- 图片显示
- 中文字库
- 菜单翻页系统
- 多页面 UI 管理器
- 动画系统
- 新硬件模块接入

---

## 5. 版本学习重点

这一版最值得学习的不是 LCD 芯片本身，而是：

### 5.1 为什么“能显示”和“显示结构清晰”是两回事

很多项目第一版显示只做到“字能出来”。  
但后面只要状态项一多，就会发现：

- 代码难改
- 页面难扩展
- 坐标和逻辑缠在一起

所以这一版要开始学：

```text
页面也需要结构设计
```

### 5.2 为什么显示服务要继续抽层

如果 `display_service_refresh_home()` 里把所有显示都写死在一个函数里，
短期能跑，但后面会越来越难维护。  

把它拆成：

- header
- 状态区
- 事件区

就是在为后续版本铺路。

### 5.3 为什么要开始考虑局部刷新

当前整页重绘很适合入门，但后面页面复杂度上来后：

- 效率
- 闪烁
- 可维护性

都会逐渐受影响。  
这一版先不一定完整实现局部刷新，但应该开始按“可局部刷新”的结构组织代码。

## 8. 我建议的实现顺序

建议代码实现顺序如下：

1. 先整理 `display_service` 内部结构
2. 再整理页面布局常量
3. 再优化页面显示效果
4. 最后看是否需要补局部刷新入口

这样做的好处是：

- 先把结构理顺
- 再调页面
- 不会一开始就把“布局调整”和“架构改动”混在一起

---

## 9. 版本完成后的预期效果

`v1.4.1` 做完后，预期应该达到：

- LCD 首页分区更清晰
- 代码阅读更容易
- 页面后续扩展更顺手
- 显示服务更像一个真正可复用的模板

也就是说，这一版完成后，LCD 就不只是“显示文本”，而是：

```text
后续所有版本的统一状态展示窗口
```
